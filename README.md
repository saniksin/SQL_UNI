# Бази даних — практичні роботи (схема `publishing`)

Репозиторій з практичними роботами з курсу **«Бази даних»**. Предметна область — **видавнича компанія**: автори, співробітники, книги, контракти та замовлення. Усі практичні працюють над однією і тією ж схемою `publishing`, яка проєктується в практичній 3 і далі реалізовується в практичних 4–6.

Середовище — **MySQL 8.0+**, клієнт — **MySQL Workbench** (або `mysql` CLI).

## Схема `publishing`

Вісім таблиць: п'ять первинних сутностей і три асоціативні (М:N):

```
     Authors ──┐                             ┌── Orders
               ├─< AuthorBook   >── Books ──<│
   Employees ──┤                             │
               ├─< EmployeeBook >── Books ──<│── OrderItem
   Authors    <┤                             │
   Employees  <┤── Contracts                 └── Books
```

- `Authors`, `Employees` — фізичні особи, з якими видавництво взаємодіє.
- `Books` — каталог (ISBN — `UNIQUE`, `YEAR` як тип для року видання).
- `Orders` — замовлення клієнтів; статус через `ENUM('New','InProgress','Completed','Canceled')`.
- `Contracts` — контракт належить **або** автору, **або** співробітнику (два `NULL`-FK + `ContractType`-дискримінатор). Ексклюзивність власника enforced тригером у практичній 6, а не `CHECK`-обмеженням.
- `AuthorBook`, `EmployeeBook`, `OrderItem` — розв'язання М:N-зв'язків з додатковими атрибутами (`AuthorOrder`, `Task`, `Quantity`/`UnitPrice`).

Детальне обґрунтування кожного рішення (чому два nullable FK, чому тригер замість CHECK, куди винесено бізнес-правила) — у [practical_3.md](practical_3.md).

## Як запустити

**Порядок обов'язковий:** `4 → 5 → 6`. Практична 4 створює базу з нуля, практичні 5 і 6 припускають, що дані вже є.

### Через MySQL Workbench

1. `File → Open SQL Script → practical_4.sql` → `Ctrl+Shift+Enter` (Execute All).
2. Те саме для `practical_5.sql` і `practical_6.sql`.
3. У `practical_6.sql` розділ «Задача 3» (тестові `INSERT`) запускати **по одному запиту**: Workbench зупиняється на першій помилці, а тут трьома `INSERT` тригер навмисно піднімає три різних помилки.

### Через `mysql` CLI

```bash
mysql -u root -p < practical_4.sql
mysql -u root -p publishing < practical_5.sql
mysql -u root -p publishing < practical_6.sql
```

> `practical_4.sql` починається з `DROP DATABASE IF EXISTS publishing;` — якщо база вже є, вона буде перестворена.

## Практична 3 — ER-модель

**Результат у практичній 3:** ER-модель із 5 первинних і 3 асоціативних сутностей, 7 бізнес-правил, позначені кратності для всіх зв'язків. Діаграма виконана у MySQL Workbench (EER Model), попередня версія — у dbdiagram.io.

**Ключові рішення:**
- М:N-зв'язки (`Author↔Book`, `Employee↔Book`, `Book↔Order`) винесено в окремі таблиці з композитним PK — це дозволяє зберігати атрибути самого зв'язку (порядок автора, роль співробітника, кількість і ціну у замовленні).
- Контракт моделюється двома nullable-полями `AuthorID` / `EmployeeID` + дискримінатором `ContractType`, а не через спільну суперсутність `Person`. Це простіше для SQL і не потребує універсального FK.
- Частина обмежень (ексклюзивний власник контракту, узгодженість дат) — бізнес-рівневі й реалізуються тригерами, а не `CHECK`-ами в DDL.

Повний розбір — [practical_3.md](practical_3.md).

## Практична 4 — DDL + DML

**Файл:** `practical_4.sql`. Структура:

1. `DROP DATABASE IF EXISTS publishing;` + `CREATE DATABASE` з `utf8mb4`.
2. `CREATE TABLE` для 5 базових + 3 асоціативних таблиць, з PK, FK, `UNIQUE` на ISBN/Email, `CHECK` на кількість і ціну в `OrderItem`, `ON DELETE CASCADE` / `RESTRICT` там, де потрібно.
3. `INSERT` по 10 записів у кожну таблицю, всередині `START TRANSACTION ... COMMIT`.
4. `UPDATE` — зміна статусу замовлення, року видання, закриття контракту.
5. `DELETE` — видалення скасованого замовлення; через каскад зникає пов'язана позиція в `OrderItem`.
6. Фінальні `SELECT` з `JOIN`: книги з авторами, замовлення з підсумком, контракти з іменем власника, підрахунок рядків у кожній таблиці.

**Після виконання скрипта очікується:**

```
Authors=10  Employees=10  Books=10  Orders=9
Contracts=10  AuthorBook=10  EmployeeBook=10  OrderItem=10
```

(`Orders=9` — одне замовлення видалено; `OrderItem` залишається 10, бо одну позицію у видаленому замовленні було скомпенсовано додатковою при вставці.)

**Висновок:** DDL покриває всю ER-модель — PK, FK, унікальні й перевірочні обмеження описані декларативно. DML показує весь життєвий цикл запису: INSERT → UPDATE → DELETE з каскадом. `JOIN`-запити у фіналі підтверджують, що зв'язки між таблицями працюють коректно.

## Практична 5 — DQL

**Файл:** `practical_5.sql`. 19 основних задач плюс блок додаткових:

- **№1–2, 9–10** — базові `SELECT` з `WHERE` і `ORDER BY` (книги за жанром, автори за країною).
- **№3, 11–12** — `JOIN` через асоціативні таблиці (автор↔книга, співробітник↔книга з роллю).
- **№4, 14** — агрегати (`COUNT`, `SUM`) з `GROUP BY`: кількість книг у жанрі, топ-автори за обсягом, виручка по книгах.
- **№5, 15** — `HAVING`: книги з виручкою понад 300 / 1000.
- **№6, 7, 16** — підзапити: `IN`, `EXISTS`, `NOT EXISTS` (автори без замовлень).
- **№8, 19** — віконні функції `RANK()` і `DENSE_RANK()` з `PARTITION BY Genre` — рейтинг книг у межах жанру.
- **№13** — позиції замовлення з `LineTotal = Quantity × UnitPrice` + агрегований `OrderTotal` по замовленню.
- **№17** — фільтр за періодом (`BETWEEN`) та списком статусів (`IN`).
- **№18** — `LEFT JOIN` з двома таблицями (`Authors`, `Employees`) через nullable FK у `Contracts`.

**Додаткові:** книги вище середнього за виручкою, автор-лідер за доходом, активні контракти (`EndDate IS NULL OR EndDate >= CURDATE()`), топ-5 жанрів за кількістю проданих екземплярів.

**Висновок:** покрито весь базовий набір DQL для MySQL 8 — від простих `WHERE`-вибірок до віконних функцій і CTE. Запити розкривають бізнес-сенс схеми: виручка по книгах/жанрах, топ-автори, активні контракти — тобто те, що видавництво реально хоче бачити у звітах.

## Практична 6 — Тригери

**Файл:** `practical_6.sql`. На таблиці `Contracts` створено два тригери, які перевіряють три бізнес-правила:

| Правило | Як перевіряється |
|---|---|
| Рівно один з `AuthorID` / `EmployeeID` заповнений | `(AuthorID IS NULL) <> (EmployeeID IS NULL)` |
| `ContractType` відповідає власнику | `AuthorID → 'Author'` / `EmployeeID → 'Employee'` |
| `EndDate >= StartDate` (якщо задано) | `EndDate IS NULL OR EndDate >= StartDate` |

Порушення будь-якого правила — `SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '...'`. Тригери потрібні у двох варіантах — `BEFORE INSERT` (`trg_contracts_bi`) і `BEFORE UPDATE` (`trg_contracts_bu`): MySQL не має «BEFORE INSERT OR UPDATE», тому логіка фактично дублюється.

**Тестові `INSERT`-и** (у файлі — задача 3):

| Сценарій | Очікуване поводження |
|---|---|
| Коректна вставка (`AuthorID=1, 'Author', 2025-06-01 → 2025-12-31`) | Успіх, запис збережено. |
| Два власники (`AuthorID=1, EmployeeID=1`) | Помилка `Exactly one of AuthorID or EmployeeID must be set`. |
| Невірний тип (`AuthorID=1, ContractType='Employee'`) | Помилка `ContractType must match owner (Author/Employee)`. |
| `EndDate < StartDate` | Помилка `EndDate must be >= StartDate`. |

**Висновок за результатами перевірки:**
1. Коректна вставка проходить — тригер не заважає валідним даним.
2. Спроба задати двох власників блокується — правило «один контракт → один власник» захищено.
3. Неузгоджений `ContractType` блокується — дискримінатор завжди синхронний із FK.
4. `EndDate < StartDate` блокується — часова логіка контрактів гарантована.

Виносити такі перевірки на рівень застосунку ризиковано: будь-який прямий доступ до БД (DBA-скрипт, інший сервіс, репліка) обійшов би правила. Тригер — єдине місце, де бізнес-логіка точно виконується.

## Технічні примітки

- **Версія MySQL.** Віконні функції (`RANK`, `DENSE_RANK`, `OVER (PARTITION BY ...)`) і CTE (`WITH ... AS`) вимагають MySQL 8.0+. На 5.7 запити з практичної 5 не спрацюють.
- **ENUM у MySQL.** Для `Role`, `Task`, `Status`, `ContractType` використано `ENUM` замість окремих довідників — це компроміс між простотою й нормалізацією. Для великого проєкту краще винести в lookup-таблиці.
- **`CHECK` vs тригери.** У MySQL 8 `CHECK` нарешті працює, але для складнішої логіки (як ексклюзивний власник контракту з перевіркою `ContractType`) тригер читабельніший і дає змістовні повідомлення помилок через `SIGNAL`.
- **Каскадне видалення.** `ON DELETE CASCADE` стоїть на асоціативних таблицях (`AuthorBook`, `EmployeeBook`, `OrderItem`). На `Contracts` — `ON DELETE RESTRICT`, бо видалення автора/співробітника з активним контрактом потребує явного закриття.
