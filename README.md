## База даних

Схема `publishing` моделює видавничий бізнес і містить **8 таблиць**:

| Таблиця | Тип | Призначення |
|---|---|---|
| `Authors` | базова | Автори книг |
| `Employees` | базова | Співробітники видавництва (редактори, коректори, перекладачі, дизайнери) |
| `Books` | базова | Каталог книг (ISBN — унікальний) |
| `Orders` | базова | Замовлення клієнтів |
| `Contracts` | базова | Контракти авторів і співробітників (один власник через `AuthorID` або `EmployeeID`) |
| `AuthorBook` | асоціативна M:N | Зв'язок автор ↔ книга |
| `EmployeeBook` | асоціативна M:N | Зв'язок співробітник ↔ книга з роллю у проєкті |
| `OrderItem` | асоціативна M:N | Позиції замовлення (книга, кількість, ціна) |

ER-модель побудована в MySQL Workbench; детальний опис — у [practical_3.md](practical_3.md).

**Файл:** [practical_3.md](practical_3.md)

**Тема:** Аналіз предметної області та побудова концептуальної моделі бази даних видавничої компанії.

**Етапи:**

1. Визначення сутностей, атрибутів та типів зв'язків (1:M / M:N).
2. Розв'язання M:N через асоціативні таблиці (`AuthorBook`, `EmployeeBook`, `OrderItem`).
3. Побудова ER-діаграми в **MySQL Workbench** (EER Model).

**Результат:** 8 сутностей (5 первинних + 3 асоціативні), 7 бізнес-правил, повна ER-діаграма — див. `practical_3.md`.

### Практична 6 — Складні SQL-вирази (тригери)

**Файл:** [practical_6.sql](practical_6.sql)

**Тема:** Тригери для валідації бізнес-логіки на рівні БД.

**Задачі:**

| # | Зміст |
|---|---|
| 1 | Тригер `trg_contracts_bi` (`BEFORE INSERT`) — валідація нових контрактів |
| 2 | Тригер `trg_contracts_bu` (`BEFORE UPDATE`) — ті самі правила при оновленні |
| 3 | Тестові `INSERT`: 1 коректний + 3 навмисні помилки |
| 4 | `SELECT`-перевірка актуальних контрактів |

**Логіка тригерів (обидва):**
- Рівно один з `AuthorID` / `EmployeeID` має бути заповнений.
- `ContractType` має відповідати власнику (`Author` або `Employee`).
- `EndDate` не може бути раніше `StartDate`.

У разі порушення будь-якого правила тригер сигналізує помилку `SIGNAL SQLSTATE '45000'` із зрозумілим повідомленням, і операція відхиляється.

**Порядок виконання:**

> **Увага:** задачу 3 запускати **по одному `INSERT` за раз** — Workbench зупиняється на першій помилці й далі не виконує наступні команди.

```
1. Виконати Задачу 1 (trg_contracts_bi) — окремим блоком
2. Виконати Задачу 2 (trg_contracts_bu) — окремим блоком
3. Задача 3 — по одному INSERT:
   a. Коректний INSERT → успішне збереження
   b. Два власники → 'Exactly one of AuthorID or EmployeeID must be set'
   c. Невірний ContractType → 'ContractType must match owner (...)'
   d. EndDate < StartDate → 'EndDate must be >= StartDate'
4. Виконати Задачу 4 (SELECT перевірка)
```

**Висновки (результати перевірки):**
1. **Коректна вставка** — запит виконано успішно, запис додано до таблиці `Contracts` без помилок. Тригер дозволяє збереження даних, які відповідають усім бізнес-правилам (один власник, правильний тип контракту, коректні дати).
2. **Помилка «два власники»** — операція відхилена тригером, оскільки одночасно вказано `AuthorID` та `EmployeeID`. Це порушує правило, що контракт може належати лише одному власнику.
3. **Помилка «неправильний тип контракту»** — вставка заблокована, бо `ContractType` не відповідає типу власника.
4. **Помилка «неправильні дати»** — запит не виконано через порушення логіки дат (`EndDate < StartDate`).

Використання тригерів `BEFORE INSERT` / `BEFORE UPDATE` дозволяє винести контроль бізнес-правил безпосередньо на рівень БД, незалежно від застосунку. Це гарантує цілісність даних навіть за помилок у клієнтському коді або прямого доступу до БД через інші інструменти.
