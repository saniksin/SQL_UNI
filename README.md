# Бази даних — практичні роботи (схема `publishing`)

Репозиторій з практичними роботами з курсу **«Бази даних»**. Предметна область — **видавнича компанія**: автори, співробітники, книги, контракти та замовлення. Усі практичні працюють над однією і тією ж схемою `publishing`, яка проєктується в практичній 3 і далі реалізовується в практичних 4–8.

Середовище — **MySQL 8.0+**, клієнт — **MySQL Workbench** (або `mysql` CLI).

## Схема `publishing`

Вісім таблиць: п'ять первинних сутностей і три асоціативні (М:N):

```
     Authors ──┐                             ┌── Orders
               ├─< AuthorBook   >── Books ──<│
   Employees ──┤                             │
               ├─< EmployeeBook >── Books ──<│── OrderItem
   Authors    <┤                             │
   Employees  <┤── Contracts                 └── Books
```

- `Authors`, `Employees` — фізичні особи, з якими видавництво взаємодіє.
- `Books` — каталог (ISBN — `UNIQUE`, `YEAR` як тип для року видання).
- `Orders` — замовлення клієнтів; статус через `ENUM('New','InProgress','Completed','Canceled')`.
- `Contracts` — контракт належить **або** автору, **або** співробітнику (два `NULL`-FK + `ContractType`-дискримінатор). Ексклюзивність власника enforced тригером у практичній 6, а не `CHECK`-обмеженням.
- `AuthorBook`, `EmployeeBook`, `OrderItem` — розв'язання М:N-зв'язків з додатковими атрибутами (`AuthorOrder`, `Task`, `Quantity`/`UnitPrice`).

Детальне обґрунтування кожного рішення (чому два nullable FK, чому тригер замість CHECK, куди винесено бізнес-правила) — у [practical_3.md](practical_3.md).

## Практична 3 — ER-модель

**Результат у практичній 3:** ER-модель із 5 первинних і 3 асоціативних сутностей, 7 бізнес-правил, позначені кратності для всіх зв'язків. Діаграма виконана у MySQL Workbench (EER Model), попередня версія — у dbdiagram.io.

**Ключові рішення:**
- М:N-зв'язки (`Author↔Book`, `Employee↔Book`, `Book↔Order`) винесено в окремі таблиці з композитним PK — це дозволяє зберігати атрибути самого зв'язку (порядок автора, роль співробітника, кількість і ціну у замовленні).
- Контракт моделюється двома nullable-полями `AuthorID` / `EmployeeID` + дискримінатором `ContractType`, а не через спільну суперсутність `Person`. Це простіше для SQL і не потребує універсального FK.
- Частина обмежень (ексклюзивний власник контракту, узгодженість дат) — бізнес-рівневі й реалізуються тригерами, а не `CHECK`-ами в DDL.

## Практична 4 — DDL + DML

**Файл:** `practical_4.sql`. Задачі за зошитом:

1. **Задача 1** — опис структури таблиць (винесено в [practical_3.md](practical_3.md), розділ «Атрибути сутностей і ключі»).

3. **Задача 3** — INSERT по 10 записів у кожну з 8 таблиць. Для асоціативних таблиць та `Contracts` / `OrderItem` використовується `INSERT ... SELECT` з пошуком батьківського рядка за унікальним полем (Email / ISBN / ClientName + OrderDate) — це робить скрипт незалежним від порядку `AUTO_INCREMENT`.
4. **Задача 4** — `SELECT *` по кожній з таблиць для візуальної перевірки вмісту.
5. **Задача 5** — quick counts (`COUNT(*)` через `UNION ALL`): очікуємо по 10 у кожній таблиці.
6. **Додатково** (не в задачах зошита) — приклади `UPDATE` (зміна статусу замовлення, року видання, закриття контракту) та `DELETE` з каскадним ефектом.

**Після виконання скрипта очікується:**

```
Authors=10  Employees=10  Books=10  Orders=10
Contracts=10  AuthorBook=10  EmployeeBook=10  OrderItem=10
```

(Задача 5 виконується до блоку «Додатково». Після `UPDATE` і `DELETE` у блоці «Додатково» `Orders` стає 9, а відповідна позиція в `OrderItem` зникає каскадом.)

**Висновок:** спроєктовано та реалізовано схему `publishing` засобами SQL — 8 таблиць з PK, FK, `UNIQUE`, `CHECK` і правилами каскадного видалення. Для асоціативних таблиць та `Contracts` / `OrderItem` застосовано `INSERT ... SELECT` з пошуком по Email / ISBN, що робить скрипт незалежним від `AUTO_INCREMENT` і безпечним для повторного запуску. Задачі 4–5 підтвердили очікувану кількість рядків у кожній таблиці; блок «Додатково» продемонстрував `UPDATE` та каскадний `DELETE`.

## Практична 5 — DQL

**Файл:** `practical_5.sql`. 19 задач за зошитом — частина задач містить по 2–3 окремих запити:

- **№1** — три варіанти простого `SELECT` (усі автори, автори з України, книги за роком).
- **№2** — `JOIN` автори ↔ книги через `authorbook`.
- **№3, 10** — `WHERE` + `ORDER BY` (книги жанру Technology; автори з України — два підзапити).
- **№4, 14** — агрегати (`COUNT`, `SUM`) з `GROUP BY`: кількість книг у жанрі, топ-автори, виручка по книгах (задача 14 містить 2 запити).
- **№5, 15** — `HAVING` (фільтр по агрегату): виручка > 1000 / > 300.
- **№6, 7, 16** — підзапити: `IN`, `EXISTS`, `NOT EXISTS` (автори без замовлень).
- **№8, 19** — віконні функції `RANK()` і `DENSE_RANK()` з `PARTITION BY Genre`.
- **№9** — базові вибірки (3 запити: співробітники, автори, книги).
- **№11** — головний автор кожної книги (через `AuthorOrder = 1`).
- **№12** — `JOIN` співробітники ↔ книги з роллю (`Task`).
- **№13** — позиції замовлень і підсумок по замовленню (2 запити).
- **№17** — `BETWEEN` + `IN`: замовлення за травень зі статусами `New` / `Completed`.
- **№18** — `LEFT JOIN` з двома таблицями (`Authors`, `Employees`) через nullable FK у `Contracts`.

**Висновок:** на схемі `publishing` відпрацьовано весь базовий набір DQL MySQL 8 — прості вибірки, `JOIN` через асоціативні таблиці, агрегати з `GROUP BY` та `HAVING`, підзапити (`IN`, `EXISTS`, `NOT EXISTS`), віконні функції `RANK` / `DENSE_RANK`. Задачі 13 і 14 показують типову пару «деталізація + агрегат» для бізнес-звітів. Запити розкривають бізнес-сенс схеми: виручка по книгах і жанрах, топ-автори, активні контракти, аналіз замовлень за періодом і статусом — тобто те, що видавництво реально хоче бачити у звітах.

## Практична 6 — Тригери

**Файл:** `practical_6.sql`. Задачі за зошитом:

- **Задача 1** — створення двох тригерів (`trg_contracts_bi` BEFORE INSERT + `trg_contracts_bu` BEFORE UPDATE) в одному блоці. Перевіряє три бізнес-правила:

  | Правило | Як перевіряється |
  |---|---|
  | Рівно один з `AuthorID` / `EmployeeID` заповнений | `(AuthorID IS NULL) <> (EmployeeID IS NULL)` |
  | `ContractType` відповідає власнику | `AuthorID → 'Author'` / `EmployeeID → 'Employee'` |
  | `EndDate >= StartDate` (якщо задано) | `EndDate IS NULL OR EndDate >= StartDate` |

  Порушення будь-якого правила — `SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '...'`.

- **Задача 2** — альтернативна детальніша версія `trg_contracts_bi` з покроковими коментарями до кожного з трьох кроків перевірки (та сама логіка, що й у задачі 1, але розгорнута).

- **Задача 3** — тестові `INSERT`-и:

  | Сценарій | Очікуване поводження |
  |---|---|
  | Коректна вставка (`AuthorID=1, 'Author', 2025-06-01 → 2025-12-31`) | Успіх, запис збережено. |
  | Два власники (`AuthorID=1, EmployeeID=1`) | Помилка `Exactly one of AuthorID or EmployeeID must be set`. |
  | Невірний тип (`AuthorID=1, ContractType='Employee'`) | Помилка `ContractType must match owner (Author/Employee)`. |
  | `EndDate < StartDate` | Помилка `EndDate must be >= StartDate`. |

- **Задача 4** — `SELECT` актуальних контрактів із сортуванням за `StartDate DESC`.

**Висновки студента** (перенесені в `practical_6.sql` після кожної задачі):

- *Задача 1.* Створено тригери для `Contracts`, які забезпечують наявність лише одного власника контракту, відповідність типу контракту його власнику та контроль дат. Механізм `SIGNAL SQLSTATE` блокує некоректні операції автоматично.
- *Задача 2.* Розгорнутий варіант `BEFORE INSERT` з покроковими коментарями — умовні оператори + `SIGNAL SQLSTATE` відхиляють некоректні дані на рівні БД, підвищуючи цілісність інформації.
- *Задача 3.* Коректна вставка проходить; спроба двох власників, неправильного типу контракту та `EndDate < StartDate` блокуються тригером — правила захищено.
- *Задача 4.* Список актуальних контрактів із сортуванням за `StartDate DESC` — найновіші першими. Отримані дані підтверджують коректність роботи БД після застосування тригерів.

## Практична 7 — Вкладені запити та повторне використання коду

**Файл:** `practical_7.sql`. Задачі за зошитом:

- **Задача 1** — вкладений запит з `NOT EXISTS` для пошуку авторів, чиї книги жодного разу не з'являлися у позиціях замовлень. Підзапит для кожного автора перевіряє наявність зв'язку `AuthorBook` ↔ `OrderItem`.
- **Задача 2** — підзапит у секції `HAVING`: книги з виручкою вище середнього рівня позиції замовлення (`AVG(Quantity * UnitPrice)` як скалярний підзапит-поріг).
- **Задача 3** — CTE (`WITH sales AS (...)`) разом з віконною функцією `RANK() OVER (PARTITION BY Genre ORDER BY Revenue DESC)`. Демонструє декомпозицію складного запиту через проміжний набір даних.
- **Задача 4** — `CREATE OR REPLACE VIEW v_book_sales` з `LEFT JOIN` (щоб у поданні з'явилися й книги без жодного продажу, з `Revenue = NULL`). Подальший `SELECT * FROM v_book_sales ORDER BY Revenue DESC` показує використання VIEW як звичайної таблиці.
- **Задача 5 / Загальний висновок** — порівняння вкладених запитів і подань зі простими `SELECT`: декомпозиція складної логіки, повторне використання, локалізація змін у визначенні VIEW.

**Висновки студента** (перенесені в `practical_7.sql` після кожної задачі):

- *Задача 1.* Складено вкладений запит з `NOT EXISTS`, який повертає авторів без жодного продажу. `NOT EXISTS` явно відповідає на запитання «чи існує замовлення для книги цього автора?» — читабельніше за `LEFT JOIN ... IS NULL`.
- *Задача 2.* Побудовано вкладений запит з `HAVING`, де скалярний підзапит обчислює «динамічний поріг» (середнє значення позиції замовлення). Така конструкція автоматично адаптується при зміні даних — на відміну від хардкодної константи.
- *Задача 3.* Реалізовано аналітичний запит з CTE та віконною функцією `RANK()`. CTE винесено окремо проміжне обчислення виручки по книзі, основний `SELECT` виконує ранжування в межах жанру через `PARTITION BY Genre` — типове рішення для задач «топ-N у категорії».
- *Задача 4.* Створено віртуальне подання `v_book_sales`, яке зберігає підрахунок виручки по книгах. Подання використовується у наступних запитах як звичайна таблиця, що зменшує дублювання коду та локалізує зміни формули виручки в одному місці.
- *Задача 5 / Загальний.* На прикладі бази `publishing` продемонстровано бізнес-сенс конструкцій повторного використання: пошук авторів без продажів, динамічний поріг «вище середнього», рейтинги книг у межах жанру, централізоване джерело даних для звітів про виручку.

## Практична 8 — Додаткові вбудовані SQL функції

**Файл:** `practical_8.sql`. Задачі за зошитом:

- **Задача 1** — текстові функції: `UPPER(Name)` (імена авторів у верхньому регістрі), `CONCAT(Name, ' <', Email, '>')` (e-mail-підпис співробітника), `LIKE '%pub.ch%'` (фільтр за корпоративним доменом), `LENGTH(Title)` (довжина назви книги в байтах).
- **Задача 2** — числові функції: `ROUND(SUM(Quantity * UnitPrice), 2)` з `GROUP BY OrderID` (загальний дохід по замовленнях), `ROUND(AVG(UnitPrice), 2)` (середня ціна книги у продажах), `MOD(Quantity, 2)` (мітка непарної кількості).
- **Задача 3** — часові функції: `CURDATE()` (поточна дата), `DATEDIFF(CURDATE(), OrderDate)` з фільтром `> 100` (старі замовлення), `YEAR(StartDate)` / `MONTH(StartDate)` (помісячно-річне представлення дат контрактів).
- **Задача 4** — логічні / умовні функції: `IF(EndDate IS NULL, 'Active', 'Closed')` (статус контракту) і `CASE WHEN PublishYear >= 2025 THEN 'Нові видання' WHEN PublishYear BETWEEN 2020 AND 2024 THEN 'Сучасні' ELSE 'Архів' END` (категоризація книг за роком).
- **Задача 5** — службові функції: `IFNULL(Phone, '— не вказано —')` (заміна NULL у телефоні автора) та `IF(Email LIKE '%@%', 'Valid email', 'Invalid email')` (базова перевірка формату e-mail).

**Висновки студента** (перенесені в `practical_8.sql` після кожної задачі):

- *Задача 1.* Відпрацьовано `UPPER`, `CONCAT`, `LIKE`-фільтр за підрядком та `LENGTH` — типові інструменти для уніфікованого виведення, формування підписів і валідації довжини рядкових полів.
- *Задача 2.* `ROUND` дала грошовий формат (2 знаки після коми) для `SUM` і `AVG`, `MOD(x, 2)` — класичний прийом перевірки парності значення безпосередньо у SELECT.
- *Задача 3.* `CURDATE`, `DATEDIFF`, `YEAR`, `MONTH` показано як основу аналітики за часовими інтервалами — фільтрація за давністю та витяг компонентів дати без додаткових полів у схемі.
- *Задача 4.* `IF` використано для бінарного статусу контракту, `CASE WHEN` — для трьох категорій книг за роком видання; обидві конструкції переносять бізнес-класифікацію в рівень запиту.
- *Задача 5.* `IFNULL` з заглушкою «— не вказано —» робить звіт читабельним, `IF(Email LIKE '%@%')` — приклад формальної перевірки e-mail на рівні SQL.

**Загальний висновок:** на схемі `publishing` відпрацьовано п'ять груп вбудованих функцій MySQL — текстові, числові, часові, логічні / умовні та службові. Перевага в тому, що форматування, агрегація, обчислення дат, класифікація та обробка NULL виконуються безпосередньо на сервері БД, без передачі сирих даних у застосунок.

## Технічні примітки

- **Версія MySQL.** Віконні функції (`RANK`, `DENSE_RANK`, `OVER (PARTITION BY ...)`) і CTE (`WITH ... AS`) вимагають MySQL 8.0+. На 5.7 запити з практичних 5 і 7 не спрацюють.
- **ENUM у MySQL.** Для `Role`, `Task`, `Status`, `ContractType` використано `ENUM` замість окремих довідників — це компроміс між простотою й нормалізацією. Для великого проєкту краще винести в lookup-таблиці.
- **`CHECK` vs тригери.** У MySQL 8 `CHECK` нарешті працює, але для складнішої логіки (як ексклюзивний власник контракту з перевіркою `ContractType`) тригер читабельніший і дає змістовні повідомлення помилок через `SIGNAL`.
- **Каскадне видалення.** `ON DELETE CASCADE` стоїть на асоціативних таблицях (`AuthorBook`, `EmployeeBook`, `OrderItem`). На `Contracts` — `ON DELETE RESTRICT`, бо видалення автора/співробітника з активним контрактом потребує явного закриття.
